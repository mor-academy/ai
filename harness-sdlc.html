05 · KB3 · Refactor lớnHarness for SDLC
Phối hợp 8 người · Tránh dẫm chân

Worktree + file-lock + PR queue — 8 task song song không conflict

1 · Worktree per task

Mỗi ticket = 1 git worktree riêng trong ~/work/<repo>-<ticket>. Branch riêng, node_modules share qua pnpm. AI chạy hoàn toàn trong worktree → main checkout của dev không bao giờ bị "dở dang".

$ git worktree add ../web-412 -b feat/web-412 $ git worktree add ../web-419 -b feat/web-419 $ git worktree list ~/acme [main] ~/work/web-412 [feat/web-412] ~/work/web-419 [feat/web-419]
2 · CODEOWNERS + auto-assign

File .github/CODEOWNERS map module → owner. PR mở → auto-request review đúng người. AI cũng đọc CODEOWNERS để biết "khi đụng auth/ phải tag @huy".

# CODEOWNERS /app/orders/ @linh @tu /app/payments/ @hung /packages/auth/ @huy @lead /db/migrations/ @lead # ai cũng phải hỏi Lead
3 · File-lock warning bot

GitHub Action chạy mỗi khi PR mở/update: so file đụng với các PR open khác. Overlap → comment cảnh báo + tag 2 owner. AI agent cũng check trước khi bắt đầu task: "có ai đang sửa file này không?" — qua gh pr list --search.

4 · Merge queue (GitHub native)

Bật merge_queue trên main: PR approved → vào queue → GitHub tự rebase + chạy CI lần cuối → merge nếu xanh. Không còn race "2 PR merge cùng lúc làm vỡ main".

5 · Shared session handoff (cross-timezone)

Linh (VN) hết giờ, task chưa xong. Gõ /handoff → AI export session summary + state + open question vào Linear comment. Sáng hôm sau Hùng (US) mở Linear, "/resume WEB-412" → AI khôi phục context, tiếp tục.

6 · Migration freeze window

File db/migrations/ chỉ Lead approve. AI có rule trong CLAUDE.md: "nếu task cần migration mới → STOP, mô tả migration cần thiết và đợi Lead approve trước khi tạo file". Tránh 2 dev cùng tạo migration trùng số.

06 · Phối hợp 8 ngườiHarness for SDLC
PR review · 4 gate, ~5 phút tới human

Cụ thể từng bot làm gì, comment trông như thế nào

~30sCI
GitHub Actions chạy tc · lint · vitest · build. Fail → bot comment đoạn log lỗi (không phải toàn bộ stdout) + suggest fix. Status check đỏ.
~2 phútCodeRabbit
CodeRabbit đọc diff → summary PR (3 bullet) + inline comment cho từng file. Tập trung: correctness, complexity, missing test. Comment có icon mức độ: 💡 nit / ⚠ warn / 🛑 block.
~3 phútSecurity bot
Semgrep chạy ruleset OWASP + custom rule team. Claude Code Action review song song: focus secret leak, IDOR, SQLi qua Drizzle, unsafe dangerouslySetInnerHTML. Comment ở line cụ thể.
~4 phútConvention bot
Action custom: đọc CLAUDE.md + diff → flag vi phạm naming/layering. Vd. "file mới ở _actions/ nhưng không export 'use server'".
~5 phútHuman
Reviewer (theo CODEOWNERS) nhận notification. Mở PR thấy: summary đã có, nit/warn đã được tác giả address, chỉ còn design/business decision cần xem. Comment depth tăng, time-to-approve giảm.
Ví dụ comment thật từ CodeRabbit
# app/_actions/list-orders.ts:42 ⚠ Potential N+1 query for (const order of orders) { order.items = await db.query .items.findMany({ where: eq(items.orderId, order.id) // ← in loop }); } Suggestion: dùng inArray để batch fetch: const items = await db.query .items.findMany({ where: inArray(items.orderId, orders.map(o => o.id)) }); # Confidence: high · Severity: warn

Tác giả PR có thể trả lời: "@coderabbit fix" → bot tự commit fix lên PR. Hoặc "@coderabbit ignore" + lý do → comment dismiss, audit log lưu.

07 · Code review pipelineHarness for SDLC
Testing · 3 playbook cụ thể

Lệnh gì, file gì, lúc nào — không lý thuyết

A · TDD-AI cho feature mới

Trước khi cho AI viết code: "Đọc acceptance criteria Linear, viết vitest test trước, chưa cần code chạy được."

$ claude > /tdd WEB-412 ✓ Tạo orders.spec.ts: - filter by single status - filter by multi status - empty filter = all - invalid status = 400 ✗ All red (chưa có impl) > Viết impl để pass test ✓ pnpm test orders → 4/4 green

→ AI có spec rõ ràng → ít bịa.

B · Property-based cho logic phức tạp

Hàm calculateDiscount(order, coupons[]): thay vì 10 test case cứng, dùng fast-check sinh ngẫu nhiên.

test("discount ≤ subtotal", () => { fc.assert(fc.property( orderArb(), couponsArb(), (o, cs) => { const d = calculateDiscount(o, cs); return d >= 0 && d <= o.subtotal; } )); });

→ Bắt edge case AI không nghĩ ra.

C · Eval cho feature dùng LLM

Feature "Tóm tắt đơn hàng" dùng Claude. Test thường không đủ — cần promptfoo.

# promptfooconfig.yaml tests: - vars: order: "@fixtures/order-01.json" assert: - type: contains-json - type: llm-rubric value: "mention total & status" - type: latency threshold: 3000 # CI: promptfoo eval --output html # PR fail nếu pass rate < 95%

→ Regression cho LLM giống unit test cho code.

Flake quarantine — autonomous

Test fail intermittently? GitHub Action quarantine-flaky chạy hằng đêm: rerun fail test 5 lần, nếu ≥ 2 pass → đánh dấu test.skip.flaky + tạo Linear ticket. CI không block, nhưng có hàng đợi để fix.

Mutation testing weekly

Stryker chạy mỗi Chủ nhật trên module quan trọng (auth, payment). Mutation score < 75% → ticket auto cho team owner: "test coverage 100% nhưng kill rate thấp, hãy bổ sung assertion thật."
